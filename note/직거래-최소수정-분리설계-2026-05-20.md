# 직거래 최소수정 분리 설계안 <!-- 2026-05-20 -->

이 문서는 현재 백엔드/프론트 구조를 최대한 유지하면서, `DIRECT` 거래를 `DELIVERY`와 체감상 다른 흐름으로 운영하기 위한 최소수정 설계안이다.

목표는 아래 3가지다.

1. 백엔드 enum, 테이블, DTO 구조 변경을 최대한 피한다.
2. 직거래는 결제/배송 흐름과 분리된 것처럼 보이게 만든다.
3. 운영 품질이 떨어지지 않도록 결제/배송 진입은 명확히 차단한다.

---

## 1. 현재 구조 요약

### 백엔드

- 거래 방식은 `TradeDeliveryType`으로 구분한다.
  - `DIRECT`
  - `DELIVERY`
- 거래 상태는 공통 `TradeStatus`를 사용한다.
  - `REQUESTED`
  - `ACCEPTED`
  - `PAID`
  - `IN_PROGRESS`
  - `RECEIVED`
  - `CONFIRMED`
  - `COMPLETED`
  - `CANCELED`
  - `DISPUTED`

관련 파일:

- `C:\dev\RE_FORM_Shop_2605\src\main\java\com\re_form_shop_2605\entity\trade\Trade.java`
- `C:\dev\RE_FORM_Shop_2605\src\main\java\com\re_form_shop_2605\entity\Enum\TradeStatus.java`
- `C:\dev\RE_FORM_Shop_2605\src\main\java\com\re_form_shop_2605\entity\Enum\TradeDeliveryType.java`
- `C:\dev\RE_FORM_Shop_2605\src\main\java\com\re_form_shop_2605\service\trade\TradeServiceImpl.java`
- `C:\dev\RE_FORM_Shop_2605\src\main\java\com\re_form_shop_2605\service\payment\PaymentService.java`

### 프론트

- `deliveryType`과 `tradeStatus`는 이미 대부분의 거래 화면에서 내려받고 있다.
- 다만 현재 `ACCEPTED`를 거의 전부 `결제 대기` 의미로 해석하고 있다.

관련 파일:

## 2. 핵심 전략

새 상태를 만들지 않는다.

대신 `deliveryType === DIRECT`일 때 기존 상태의 의미를 다르게 해석한다.

### 직거래 상태 의미 재해석

| 실제 상태값 | 택배 거래 의미 | 직거래 의미 |
|---|---|---|
| `REQUESTED` | 구매 요청됨 | 직거래 요청됨 |
| `ACCEPTED` | 결제 대기 | 채팅 조율 중 |
| `CONFIRMED` | 구매 확정 | 직거래 완료 확인 |
| `COMPLETED` | 거래 완료 | 거래 완료 |
| `CANCELED` | 거래 취소 | 거래 취소 |

### 직거래 목표 흐름

`REQUESTED -> ACCEPTED -> CONFIRMED -> COMPLETED`

사용자 체감 기준:

- 구매자가 판매글에서 직거래 요청
- 판매자가 수락
- 채팅에서 장소/시간/상품 상태 조율
- 거래가 끝나면 구매자가 완료 처리
- 기존 정산 완료 시점 로직은 그대로 `COMPLETED`

## 3. 백엔드 최소수정안

### 3-1. 꼭 수정해야 하는 부분

#### A. 직거래는 `ACCEPTED`에서도 구매확정 가능하게 허용

현재 `confirm()`은 `PAID`, `IN_PROGRESS`, `RECEIVED`에서만 동작한다.

수정 대상:

- `C:\dev\RE_FORM_Shop_2605\src\main\java\com\re_form_shop_2605\entity\trade\Trade.java`
- `C:\dev\RE_FORM_Shop_2605\src\main\java\com\re_form_shop_2605\service\trade\TradeServiceImpl.java`

수정 방향:

- `deliveryType == DIRECT && status == ACCEPTED`면 `confirm()` 허용
- `TradeServiceImpl.confirmTrade(...)`의 상태 검증도 동일 기준으로 허용

의도:

- 직거래는 결제/배송 없이도 `ACCEPTED -> CONFIRMED`로 갈 수 있게 만든다.

#### B. 수락 후 시스템 메시지를 거래방식별로 분기

현재는 수락 시 무조건 “결제를 진행해 주세요” 메시지가 발송된다.

수정 대상:

- `C:\dev\RE_FORM_Shop_2605\src\main\java\com\re_form_shop_2605\service\trade\TradeServiceImpl.java`

수정 방향:

- `DELIVERY`: 기존 메시지 유지
- `DIRECT`: `판매자가 거래를 수락했습니다. 채팅에서 장소와 시간을 조율해 주세요.`

의도:

- 사용자가 직거래인데 결제 단계로 오해하지 않게 만든다.

### 3-2. 강력 권장 수정

#### C. 직거래는 결제 API 진입을 명시적으로 차단

현재 결제 서비스는 `ACCEPTED` 상태면 결제 가능하고, `DELIVERY`일 때만 배송지 검증을 추가로 한다.

수정 대상:

- `C:\dev\RE_FORM_Shop_2605\src\main\java\com\re_form_shop_2605\service\payment\PaymentService.java`

수정 방향:

- `trade.getDeliveryType() == DIRECT`면 결제 생성 자체를 차단
- 에러 메시지는 `직거래는 결제를 생성할 수 없습니다.` 수준으로 명확히

의도:

- 프론트에서 버튼을 숨겨도 직접 API 호출하면 결제가 생길 수 있으므로 서버 방어가 필요하다.

### 3-3. 수정하지 않아도 되는 부분

- 새 enum 추가
- 새 테이블 추가
- `ChatRoomDetailDTO` 구조 변경
- 새 직거래 전용 엔드포인트 추가
- 정산 배치 구조 변경

## 4. 프론트 최소수정안

### 4-1. 핵심 원칙

- `DELIVERY`는 기존 UI를 최대한 유지한다.
- `DIRECT`만 상태 라벨, 안내 문구, CTA를 다르게 보인다.
- 결제/배송 관련 UI는 `DIRECT`에서 숨긴다.

### 4-2. 우선 수정 대상

#### A. 채팅방 상단 CTA/상태 뱃지

파일:

- `C:\dev\re-form_view\src\pages\chat\ChatPage.tsx`

현재 문제:

- `ACCEPTED`면 무조건 `결제하기`
- `RECEIVED`일 때만 `구매 확정`

수정 방향:

- `DIRECT + ACCEPTED`
  - 상태 라벨: `조율 중`
  - 버튼: `거래 완료하기` 또는 `거래 완료 요청`
  - 링크: `/trade/{tradeId}/confirm`
- `DIRECT`
  - `PAID`, `IN_PROGRESS` 전용 안전결제 배너 숨김

#### B. 거래 상세 페이지 메인 상태 카드

파일:

- `C:\dev\re-form_view\src\pages\trade\TradePage.tsx`

현재 문제:

- `ACCEPTED` 블록이 사실상 결제 단계 화면이다.
- `DIRECT`도 결제 흐름 안에 끌려간다.

수정 방향:

- `DIRECT + ACCEPTED + 구매자`
  - 조율 안내 카드
  - `거래 완료하기` 버튼
- `DIRECT + ACCEPTED + 판매자`
  - `구매자와 채팅으로 장소와 시간을 조율하세요`
  - 결제 대기 문구 제거
- `DIRECT`
  - 배송지 입력 UI 제거
  - 결제 버튼 제거
  - 배송지 미입력 경고 제거

#### C. 거래 확정 페이지

파일:

- `C:\dev\re-form_view\src\pages\trade\TradeConfirmPage.tsx`

수정 방향:

- `DIRECT`일 때는 `상품 수령 후 구매 확정` 표현 대신
  - `직거래가 완료되었다면 거래를 확정해 주세요`
- `DELIVERY`는 기존 문구 유지

#### D. 마이페이지 거래 상태 라벨

파일:

- `C:\dev\re-form_view\src\pages\mypage\MyPage.tsx`

수정 방향:

- `DIRECT + ACCEPTED` -> `조율 중`
- `DIRECT + CONFIRMED` -> `완료 확인`
- `DIRECT + COMPLETED` -> `거래 완료`

### 4-3. 이미 있는 API는 그대로 사용

파일:

- `C:\dev\re-form_view\src\features\trade\api\tradeApi.ts`

유지 가능한 이유:

- `acceptTrade()`
- `confirmTrade()`
- `cancelTrade()`

이 3개만으로도 직거래의 핵심 흐름은 만들 수 있다.

즉, 프론트는 새 API를 붙이지 않고도 `DIRECT` 전용 UX를 구성할 수 있다.

- `거래 완료하기`

### 완료 직전 안내

- `직거래가 끝났다면 거래를 완료 처리해 주세요. 완료 후 거래 상태가 반영됩니다.`

## 6. 단계별 구현 순서

### 1단계. 백엔드

1. `DIRECT + ACCEPTED`에서 `confirmTrade()` 허용
2. `DIRECT` 수락 후 시스템 메시지 분기
3. `DIRECT` 결제 API 차단

### 2단계. 프론트

1. `ChatPage`에서 `DIRECT + ACCEPTED` CTA 교체
2. `TradePage`에서 `DIRECT + ACCEPTED` 메인 카드 교체
3. `TradeConfirmPage` 직거래 전용 문구 조정
4. `MyPage` 상태 라벨 재해석

### 3단계. 검증

1. `DIRECT` 글 요청 후 판매자 수락
2. 채팅방에 결제 버튼이 보이지 않는지 확인
3. 구매자/판매자 모두 직거래 안내 문구가 보이는지 확인
4. 구매자가 완료 처리 시 `CONFIRMED`로 가는지 확인
5. 이후 기존 완료/정산 흐름이 유지되는지 확인

## 7. 이 안의 장점과 한계

### 장점

- 상태 enum 추가 없음
- DB 스키마 변경 없음
- 새 DTO 거의 불필요
- 프론트/백엔드 모두 수정 범위가 작다
- 사용자 체감상 직거래와 택배거래가 분명히 분리된다

### 한계

- `만남 확정` 같은 중간 단계를 별도 저장하지 않는다
- 직거래 세부 진행 상황은 채팅 맥락에 의존한다
- 아주 정교한 운영 상태 추적에는 한계가 있다

이번 목표가 `최소 수정 + 충분한 분리감`이라면, 이 안이 가장 현실적이다.
